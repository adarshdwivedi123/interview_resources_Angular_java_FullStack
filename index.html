🔐 Role-Based Routing — Interview Notes

Project: RM Tool (Angular) · Simple words, ready to explain

Say this first: "When a user logs in, the backend tells us their role. We save it. Then route guards check the role before opening any page. The sidebar menu is also built from the role, and a custom directive hides individual buttons. The backend does the real security check."

👮 The 5 Roles

district_entry district_approver spc ministry admin

🏢 Explain it like I'm 5 (the office building)

Think of the app as an office building:

  • Login = security desk gives you an ID badge with your role.
  • Route guards = door locks. Your badge only opens your doors. Wrong badge → sent back to the lobby.
  • Sidebar menu = a personalized directory — you only see floors your badge reaches.
  • Custom directive = some drawers stay locked — buttons like "Approve" only show for the right role.
  • Backend = the vault — even with a fake badge, the vault checks who you really are.

💻 The Code (memorize these 4)

1. The check — AuthService

readonly userRole = computed(() => this.currentUser()?.role ?? null);

hasRole(...roles: UserRole[]): boolean {
  const current = this.userRole();
  return current !== null && roles.includes(current);
}

2. The guard factory — auth.guard.ts

function roleGuard(...allowedRoles: UserRole[]): CanActivateFn {
  return (route, state) => {
    const auth = inject(AuthService);
    const router = inject(Router);
    if (auth.isLoggedIn() && auth.hasRole(...allowedRoles)) return true;   // allow
    if (!auth.isLoggedIn()) { router.navigate(['/login']); return false; } // not logged in
    router.navigate(['/app/dashboard']); return false;                     // wrong role
  };
}

export const adminGuard       = roleGuard('admin');
export const approverOrSpcGuard = roleGuard('district_approver', 'spc');

3. The wiring — app.routes.ts

{
  path: 'app',
  canActivate: [authGuard],                                  // must be logged in
  children: [
    { path: 'dashboard',     loadComponent: ... },           // everyone
    { path: 'admin-users',   canActivate: [adminGuard], ... },// admin only
    { path: 'pending-review',canActivate: [approverOrSpcGuard], ... }, // 2 roles
  ]
}

4. The custom directive — has-role.directive.ts NEW

@Directive({ selector: '[appHasRole]', standalone: true })
export class HasRoleDirective {
  private tpl  = inject(TemplateRef<unknown>);
  private vcr  = inject(ViewContainerRef);
  private auth = inject(AuthService);

  @Input({ required: true }) set appHasRole(roles: UserRole[]) {
    this.vcr.clear();
    if (this.auth.hasRole(...roles)) this.vcr.createEmbeddedView(this.tpl); // show
  }                                                                          // else: removed
}
<button *appHasRole="['district_approver']">Approve</button>
<button *appHasRole="['spc', 'admin']">Export Report</button>

Real usage in this project: the Documents page (documents.html) gates the "Download Roster Template" button with *appHasRole="['district_entry','spc','admin']" — so the Approver (read-only there) never sees it. The directive is added to the component's imports array.

🔄 The Flow (tell it as a story)

1
User logs in → backend sends role → saved in a signal + sessionStorage.
2
User clicks a page → Angular runs the route's guard first.
3
Guard asks AuthService hasRole(...) → true = show page, else redirect.
4
The sidebar is built from the role, so the menu matches access.
5
On shared pages, *appHasRole hides buttons the user can't use.
6
Every API call carries the token → backend does the real check.

🎤 Likely Interview Questions

Q: What if a user edits their role in sessionStorage to "admin"?

They only fake the UI. Every API call carries a signed JWT, and the backend checks the role from the token — not from what the frontend claims. Guards are UX; the server is the real security.

Q: CanActivate vs CanMatch?

CanActivate runs after the route matches — decides if you can enter. CanMatch runs during matching, can skip the route entirely and stop the lazy chunk from even downloading. We use CanActivate.

Q: Why signals instead of Observables?

Simpler and synchronous — auth.userRole() just reads the value. Guards and the menu react automatically, no subscribe/unsubscribe, no leak risk.

Q: Why sessionStorage, not localStorage?

sessionStorage is per-tab and clears on close — safer for an internal tool. A BroadcastChannel also blocks the same user running two sessions.

Q: A route for multiple roles?

The factory takes a rest param: roleGuard('district_approver','spc'). hasRole() uses includes(), so any role in the list passes.

Q: Why a directive when you already have *ngIf?

Reusable and declarative — one directive instead of repeating *ngIf="auth.hasRole(...)" everywhere. It removes the element from the DOM and reuses the same hasRole() logic, so role rules live in one place.

Q: What would you improve?

I added the *appHasRole directive for element-level control. Next: a single permissions map (role → allowed routes) as the source of truth, switch to CanMatch so unauthorized chunks never load, and a token-expiry interceptor.

Maturity line to drop in: "Frontend guards and the directive are for navigation and UX — the real security is the backend validating the token's role on every request. The frontend just shouldn't show what it can't use."
